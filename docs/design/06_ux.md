## 1. 공간 구조 — "한 씬, 다섯 스테이션, 걷기 없음"

**결론: 이동 조작 없이 상단 탭으로 공방·꾸미기·사격장·PVP·보관함 5개 스테이션을 전환한다.** 공방·꾸미기·보관함은 편집 무대를 공유하고, 사격장과 PVP는 각각 전용 3D 그룹·HUD를 켠다. 이 `StationId` 5종이 전 문서의 공간 정본이다. 사격 공간 한글명 = **"사격장"**.

| 후보 | 판정 | 이유 |
|---|---|---|
| 씬 5개 분리 + 로딩 전환 | ✗ | 씬 재구축 비용·상태 동기화 부담. 총 모델 인스턴스를 옮겨 심어야 함 |
| 1인칭 걸어서 이동 (전작 방식) | ✗ | 초등학생에게 이동 자체가 마찰. "사격장 어디야?"가 발생. 3클릭 원칙 위반 |
| **한 씬 + 편집/사격/PVP 그룹·패널 토글** | ✓ | 편집 무대는 공유하고 사격장·PVP만 전용 그룹·HUD 사용 |

- 현재 전환은 즉시 토글이며 같은 탭 재선택은 no-op이다. `STATION_DEFS.mode/panel`을 사용하므로 신규 편집 스테이션이 사격장으로 잘못 진입하지 않는다.
- 내가 만든 총은 모든 스테이션에서 같은 `Blaster` 데이터와 `buildBlaster()`를 사용해 재구성한다. 공방·뷰모델이 다른 외형으로 드리프트하지 않는다.
- 사격장만 카메라가 총 뒤(견착 시점)로 붙는 **1인칭 사대 고정**(결정문 18 — Pointer Lock 안 씀), 공방·페인트는 총을 앞에 두고 도는 궤도 카메라.
- **화면 방향은 가로(landscape) 기준** — 좌 3D 뷰 + 우 패널(결정문 22). 카메라 프리셋도 가로 FOV 기준. 모바일 세로 대응은 **M3+ 반응형 과제**.

```ts
// src/game/screens.ts (leaf — main.ts 는 wire 만)
export type StationId = "workshop" | "paint" | "range" | "pvp" | "collection";
// 실제 타입은 src/game/definitions.ts STATION_DEFS의 key에서 파생한다.

export interface StationCamera {
  pos: [number, number, number];      // 카메라 목표 위치
  lookAt: [number, number, number];   // 총 거치대 기준
  orbit: boolean;                     // 궤도 회전 허용 (workshop·paint 만 true)
}

// 전환 애니메이션은 현재 없음. 도입 시 policy에 추가하고 테스트한다.
```

## 2. 조립 UX — 탭 선택이 정답, 드래그는 보조

**결론: "슬롯 탭 → 카탈로그에서 파츠 탭 → 즉시 찰칵 부착" 2탭(클릭) 방식을 기본으로 한다.** 드래그&드롭은 보너스이지 필수 경로가 아니다(M1 이후, 실패 시 원위치 스냅백).

### 흐름 (공방 화면)
1. 총 3D 뷰에서 **파츠가 장착된 슬롯이 은은하게 빛난다**(펄스 1.2초 주기). 빈 소켓은 메시로 그리지 않고(03 "빈 소켓=없는 소켓" 표현 규칙 — 결정문 17) **UI 오버레이 점선 링**으로만 위치를 알린다. 슬롯 탭 → 해당 슬롯 확대 + 우측 카탈로그가 그 슬롯용 파츠로 필터됨.
2. 카탈로그에서 파츠 카드 탭 → **즉시 부착 + "찰칵" SFX + 파츠가 0.15초 살짝 커졌다 줄어드는 팝**. 미리보기 단계 없음 — 바꿔보는 것 자체가 재미이므로 적용을 지연시키지 않는다.
3. 같은 슬롯의 다른 파츠 탭 = 교체. 장착 중인 파츠 카드 다시 탭 = 분리. 단 **body 슬롯은 유일한 필수 슬롯이라 분리 불가, 교체만**(결정문 16) — 나머지를 다 떼면 "맨몸" 상태.
4. **파츠 선택 = 변형 진입** — 선택한 파츠의 morph 슬라이더 2~3개가 우측 패널 카탈로그 아래로 슬라이드-인(§5). 별도 "변형 모드" 없음, 조립과 변형은 같은 공방의 한 호흡. **상세 사양 정본은 09 §5.**

### 회전·탭 판별 — 시작점 무관 단일 규칙
아이는 총을 **직접 잡고** 돌리려 한다. 따라서:

- **pointerdown 위치 무관** — 이동이 **8px 를 넘으면 회전(드래그)**, 8px 이내에서 up 이면 **raycast 탭**(슬롯/파츠 선택). 이 판별은 pointer 이벤트 레이어가 담당하고, 회전·줌 자체는 **three/addons OrbitControls 정본**(결정문 24)에 병행 배선한다.
- OrbitControls damping 활성, 줌 거리 0.6x~2.0x 를 min/maxDistance 로 클램프.

### 실수 복구 — 3중 안전망
- **모든 조합이 유효하다.** "고장"·"조립 실패" 상태가 데이터 모델에 없다(02 전제). 실수해도 이상한 총이 나올 뿐, 벌은 없다.
- **되돌리기(↩) 버튼**: undo 스택은 **kind 분리**(조립 30 + 색칠 30, redo 없음 — 결정문 15). **M1 은 되돌리기 버튼 1개 — 공방의 조립 스택이 파츠 교체와 morph 변형을 통합**해 되감고(1 슬라이더 제스처 = 1 엔트리, 정본 09 §5), 색칠 스택은 paint 스테이션의 되돌리기 버튼이 pop 한다. 색칠을 12번 해도 조립 undo 가 잡아먹히지 않는다.
- **처음부터(리셋) 버튼**: 확인 다이얼로그(텍스트) 대신 **1초 꾹 누르기(hold-to-confirm)** — 버튼에 원형 게이지가 차오르고, 다 차면 스타터 구성으로 리셋. **M1 에는 이 버튼 자체를 탑재하지 않는다**(undo 30 이면 충분).

### undo 스냅샷 — Blaster(02 정본)의 부분 스냅샷
조립 데이터의 단일 타입 `Blaster` 정본은 02 다(결정문 12) — 이 문서의 구 `BlasterConfig` 는 폐기. undo 는 전체가 아닌 **부분 스냅샷**을 캡처한다:

- **조립 스택 엔트리 = `{슬롯: {partId, morph}}` — paint 를 절대 캡처하지 않는다.** 복원은 현재 paint 를 보존하는 merge 로 수행(상세 정본 09 §5 — kind 분리 취지 보호).
- **색칠 스택 엔트리 = 파츠 인스턴스별 3존 PaletteKey 맵**(결정문 11·12).
- 스냅샷 하나가 수백 바이트, 깊이 30×2 여도 수 KB — 메모리 걱정 없음. **획(stroke) 페인팅 도입은 05 세이브 스키마 확장(paintOps[])+마이그레이션이 선행 조건**(§9).

```ts
// src/game/undoStack.ts (순수 로직 — 단위테스트 대상)
import type { SlotType, PartId, PaletteKey } from "./blaster"; // 타입 정본 = 02
import type { MorphState } from "./morph";                     // 정본 = 09

export type AssemblySnapshot = Partial<Record<SlotType, { partId: PartId; morph: MorphState }>>; // paint 미포함
export type PaintSnapshot = Partial<Record<SlotType, Record<"primary" | "secondary" | "accent", PaletteKey>>>;

export type UndoKind = "assembly" | "paint";

export interface UndoStack {
  push(kind: UndoKind, snapshot: AssemblySnapshot | PaintSnapshot): void;
  undo(kind: UndoKind): AssemblySnapshot | PaintSnapshot | null; // 없으면 null — UI 는 버튼 흐리게
  size(kind: UndoKind): number;
}
export const UNDO_DEPTH_PER_KIND = 30;
```

## 3. 3클릭 원칙 — 첫 발사까지의 경로

첫 실행 시 **몸통만 장착된 "맨몸" 블래스터**로 시작한다(온보딩 정본 = 01 "맨몸 첫 사격", 결정문 16 — body 가 유일한 필수 슬롯이므로 성립). 맨몸 기본 발사값은 02 `computeStats` 가 정의한다. 타이틀에 버튼 2개만 둔다.

| 클릭 | 행동 | 화면 |
|---|---|---|
| 1 | 타이틀 `[바로 쏘기!]` 클릭 | 사격장으로 카메라 이동 (M1 코스 = "풍선 마당" 1개, 결정문 20) |
| 2 | 사격장 화면 클릭 = **발사** (§6 발사 규칙과 동일) | 다트가 포물선(04)으로 날아가 풍선 명중 |

→ 실제로는 **2클릭에 첫 발사**. `[만들기]` 버튼으로 시작한 아이도 공방→사격장 탭→발사 = 3클릭 안에 도달. 재방문(세이브 존재) 시 타이틀은 마지막 스테이션을 기억해 바로 그 자리에서 시작한다.

여기서 학습된 "탭하면 나간다"는 이후에도 **영구히 유효하다** — §6 의 발사 규칙이 동일한 탭/클릭 발사를 보장하므로, 첫 경험과 본편 조작이 어긋나지 않는다.

## 4. 텍스트 최소화 — 아이콘 + 세 글자 한글

- 모든 버튼 = **절차 SVG 아이콘(이모지 금지 — 결정문 25) + 최대 3글자 한글 라벨**. 저학년 어휘만: `쏘기`·`만들기`·`색칠`·`과녁`. "커스터마이즈"·"시뮬레이션" 같은 단어는 UI 에 등장 금지. 금칙어 정본 = 08 §3.1.
  - 3글자로 안 되는 동작(되돌리기·처음부터)은 **아이콘 단독 버튼**으로 두고, 첫 사용 시 손가락 포인터가 안내한다. 규칙과 화면이 어긋나는 4글자 라벨을 만들지 않는다. (예외: 슬라이더 양끝 라벨은 버튼이 아니라 4글자 허용 — 09 §1)
  - `저장` 은 어휘 목록에서 **삭제** — 저장 버튼이 아예 없다(§9 전부 자동저장, 결정문 14).
- **숫자 스탯 화면 노출 금지**(결정문 3 — 예외는 결과 화면 기록값뿐, 08 §6 범위 한정). 스탯은 **별 5개(★★★☆☆)** 로만 표시 — 내부 1~10 스탯의 5단 표시이며 매핑 정본은 02. M1 노출 스탯 = 파워·정확·연사(목록 정본 02; **"멋" 스탯은 삭제**·다트 수(capacity)·재장전은 M2 노출 — 결정문 3·21). 파츠 교체·변형 시 변하는 별이 **노랗게 반짝 + 위/아래 화살표 0.4초** — "이걸 끼우면 세진다"를 글 없이 전달.
- 튜토리얼 = 텍스트 패널이 아니라 **손가락 포인터 애니메이션**(다음에 누를 곳 위에서 통통 튐). 전작 `coachBeacon.ts` 의 유도 비콘 패턴 재사용.

```ts
// src/ui/statStars.ts
export function statStars(value: number, max = 5): string; // 0..5 → "★★★☆☆" DOM 생성, 변화 시 diff 애니메이션. 09 연속(부분) 채움 렌더 겸용
```

## 5. 화면 와이어프레임 — 가로(데스크톱) 기준

**결론: 3스테이션 전부 PC 데스크톱 가로 기준 — 좌 3D 뷰 + 우 패널(결정문 22).** 스케치의 이모지·별 글리프는 자리표시자이며 실제 아이콘은 절차 SVG(결정문 25).

### 공방 (조립대)
```
┌──────────────────────────────────┬───────────────────────────┐
│ [↩]                              │ [몸통][배럴][조준기]        │ ← 슬롯 필터 칩 (M2: 다트 팩 등)
│                                  │ ┌────┐┌────┐              │
│        ╔═══════════════╗         │ │파츠││파츠│  ↓ 스크롤     │ ← 카탈로그 (카드 88px, 장착 중 = 체크 배지)
│        ║    총 3D 뷰    ║         │ └────┘└────┘              │
│        ║  ◉    ◉    ◉  ║         │ ─── 변형 슬라이더 ───      │
│        ╚═══════════════╝         │ 길이    ├───●───┤          │ ← 선택 파츠 morph 2~3개
│   파워 ★★★☆☆  정확 ★★★★☆        │ 통통함  ├───●───┤          │    (배치만 여기 — 정본은 09 §5)
│   연사 ★★☆☆☆                    │ [쏴보러 가기]     ◎퍼짐 원   │ ← 사격장 바로가기 + 퍼짐 프리뷰(09 §4)
├──────────────────────────────────┴───────────────────────────┤
│            [만들기]     [색칠]     [쏘기]                       │ ← 하단 스테이션 탭 (항상 표시, 72px)
└──────────────────────────────────────────────────────────────┘
```
◉ = 장착 슬롯 펄스(빈 소켓은 점선 링 오버레이 — §2). 드래그(>8px)=회전, 휠=줌, 클릭=슬롯/파츠 선택.

### 사격장 — "풍선 마당" (결정문 20)
```
┌──────────────────────────────────────────────────────────────┐
│ 파워★3 정확★4 연사★2                       명중 4 / 9        │ ← 별 요약 1줄 + 명중 수(간이)
│          🎈        🎯          🎈                             │ ← 고정 과녁 3 + 풍선 6, 멀수록 작게
│                    ╋                                         │ ← 조준 링: 퍼짐의 화면 투영 (§7)
│                                                              │
│         ▄▄▄ 내 총 견착 뷰 (사대 고정 1인칭) ▄▄▄                │
│                                                              │
│  (화면 전체: 드래그=조준, 클릭/탭=발사)         ┌──────┐       │
│                                               │ 발사! │       │ ← 우하단 96px (클릭=단발, 화면 클릭과 동일)
│            [만들기]     [색칠]     [쏘기]       └──────┘       │
└──────────────────────────────────────────────────────────────┘
```
간이 별점(명중 수 기준)은 코스 종료 화면에서 — 정식 점수판·콤보는 M2(결정문 20). 결과 화면 기록값만 수치 허용(08 §6).

### 페인트 부스
공방과 동일 골격에서 우측 카탈로그 자리가 **팔레트(PaletteKey 스와치)** 로 바뀐다 — 자유 RGB·색상환 금지(결정문 11). **M1 도구는 채우기 1개뿐이므로 도구 선택 UI 자체가 없다** — 스와치 탭 → 3D 뷰에서 존 탭 = 그 존(primary/secondary/accent, 결정문 12)을 그 색으로 채움. base 존은 밝은 색 강제 팔레트(03·08 슬롯 차등 — 결정문 25). 브러시·스티커는 M1 이후(§6 하단 참조), 도입 시 도구 버튼 행이 추가된다. 페인트 부스의 되돌리기 버튼 = 색칠 스택 전용(§2).

## 6. 입력 스킴 — 스테이션별로 다르고, 각각은 단순하게

**M1 입력 = 마우스**(결정문 22 — 터치는 M2). 전작 `touchControls.ts` 패턴 계승: **순수 표현+입력 leaf 모듈(main.ts import 금지), 게임과는 좁은 콜백 인터페이스로만 통신**. 터치 도입 시 같은 콜백에 탭/드래그/핀치를 배선하면 된다.

**공통 판별 규칙 (전 스테이션 동일): pointerdown 후 이동 ≤8px 에서 up = "탭/클릭", >8px = "드래그".** pointer 이벤트 레이어가 판별하고 회전·줌은 OrbitControls 병행(결정문 24). 탭·드래그의 의미만 스테이션별로 달라진다.

| 스테이션 | 입력 (M1 마우스) | 동작 |
|---|---|---|
| 공방 | 클릭 (3D 뷰 = raycast, 카드/칩/슬라이더 = DOM) | 슬롯 선택·파츠 부착·morph 슬라이더(09) |
| 공방 | 드래그 (시작점 무관) | 총 궤도 회전 (OrbitControls damping) |
| 공방 | 휠 | 줌 (거리 0.6x~2.0x 클램프) |
| 페인트 | 스와치 클릭 → 존 클릭 | 존 단위 채우기 (M1 유일 도구) |
| 페인트 | 드래그 (시작점 무관) | 총 궤도 회전 — 모드 의존 없음 |
| 사격장 | **클릭/탭 (UI 히트영역 제외)** | **단발 발사** (결정문 18) |
| 사격장 | 드래그 | 조준 (사대 고정 1인칭, Pointer Lock 안 씀, 감도 낮게) |
| 사격장 | 발사 버튼 클릭 | 단발 (화면 클릭과 동일 결과 — 어느 쪽이든 나간다) |
| 사격장 | 발사 버튼 홀드 | 연사 (**M1 이후** — §7 연사 스탯 참조) |
| PVP 로비 | 보관함 카드 버튼 | 해당 블래스터를 장착하고 3라운드 시작 |
| PVP | 마우스 위치 / 좌클릭 홀드·Space | 절대 위치 조준 / 연사 간격대로 발사 |
| PVP | A/D·좌우 방향키 / Esc | 좌우 피하기 / 보관함으로 |

### 발사 입력 — 단일 규칙 (자기모순 제거)
- **사격장에서 클릭/탭(이동 ≤8px, 하단 스테이션 탭·HUD·발사 버튼 등 UI 히트영역 제외) = 단발.** §3 의 첫 발사 경험과 동일.
- **드래그 = 조준.** 드래그로 판정된 포인터는 up 시 발사하지 않는다.
- **발사 버튼 = 단발.** M1 이후 홀드 연사가 붙는다. "조준 드래그 + 홀드 연사 동시"는 두 입력 동시 조작이므로, **소프트 오토에임(조준점 근처 20px 내 타겟에 살짝 끌림) 도입 시점에 함께 해결한다** — 오토에임이 있으면 홀드 중 조준을 놓아도 맞는다. 설정의 "에임 어시스트" 토글(§10)이 이 기능의 스위치다.
- M1 은 클릭 단발만 있으므로 **한 손(마우스만) 조작 원칙이 자동으로 지켜진다.** **키보드 필수 조작 없음.**

```ts
// src/ui/stationInput.ts (leaf — M1 마우스, M2 터치를 같은 콜백에 배선)
export const TAP_SLOP_PX = 8;

export interface StationInputCallbacks {
  tapWorld(ndcX: number, ndcY: number): void;  // raycast 는 game 쪽 책임. range 에선 발사로 해석
  aim(dx: number, dy: number): void;           // range 전용
  fireOnce(): void;                            // 클릭 단발 (M1)
  setFiring(on: boolean): void;                // 홀드 연사 (M1 이후)
  activeStation(): StationId;                  // 스킴 분기 (전작 isPlaying() 패턴)
}
// 궤도 회전·줌은 OrbitControls(three/addons) 소관 — 이 콜백 밖 (결정문 24)
```

### 브러시·스티커는 M1 이후 — 기술 스파이크 선행
프리미티브 조합 절차 메시에는 저작된 UV 가 없다. 자유 획 = raycast→UV 매핑→render-to-texture 파이프라인 + 파츠 교체 시 텍스처 승계 문제가 따라오는 큰 덩어리로, 30분 증분으로 쪼개지지 않는다. 따라서:
- **M1 색칠 = 존(3존) 단위 채우기만.** 구현은 **03 의 색|finish 머티리얼 캐시 계약**(결정문 10)을 따른다 — 이 문서의 구 clone 방식 서술은 폐기, 머티리얼 예산도 03 캐시 계약 참조.
- 브러시·스티커는 **"데칼 방식 vs UV 페인팅" 스파이크 1회 후 결정** 항목으로 M1 이후 칸에 둔다.
- 브러시 도입 시 모드 의존 제스처("브러시 모드일 때만 드래그=획")를 쓰지 않는다 — 같은 손짓의 의미가 보이지 않는 상태에 따라 바뀌면 저학년이 혼란에 빠진다. 대신 **총 아래 턴테이블 핸들(가로 드래그 링)** 을 상시 제공해, 모드와 무관하게 항상 같은 자리·같은 동작으로 회전한다.

## 7. 스탯 체감 매핑 — 보여주는 방법 (수치 정본 = 04)

**별이 바뀌면 다트가 다르게 날아가야 한다.** 조립·변형→검증→개선 루프의 성패는 파츠 교체가 사격장에서 **글 없이 몸으로 느껴지는가**에 달려 있다. 단, **수치의 정본은 04 ShotProfile 계약**(탄속 20~60m/s·퍼짐 0.3~4.0°·연사 간격 150~600ms·반동 파생 — 결정문 4)이다. **이 문서의 구 독자 수치 테이블(별당 +6m/s·단계 배열 등)은 폐기** — 이 섹션은 그 수치를 **어떤 채널로 보여주는가**만 소유한다.

| 스탯 | 체감 채널 | 수치 근거 | M1 |
|---|---|---|---|
| **파워** | 눈으로 보이는 탄속 차이 | 04 탄속 매핑 (20~60m/s) | **✓** |
| **파워** | 명중 시 과녁 넉백·풍선 밀림 | 04 | M1 이후 |
| **파워** | 발사음 시작 피치 변주 (파워↑ = 피치↓ — §8) | §8 | M1 이후 |
| **정확** | 탄퍼짐 원뿔 | 04 퍼짐 매핑 (0.3~4.0°) | **✓** |
| **정확** | **조준 링 크기 = 퍼짐 원뿔의 화면 투영** — 정확이 높을수록 링이 조여진다. "링 안 어딘가에 맞는다"를 시각 계약으로 | 04 → 화면 투영 | **✓** |
| **연사** | 클릭 발사 쿨다운 — 연타 시 몸으로 느껴짐 | 04 간격 매핑 (150~600ms) | **✓** |
| **연사** | 홀드 연사 (같은 간격) | 04 | M1 이후 |
| (juice) | 명중 파티클 — **"멋" 스탯 삭제(결정문 3)로 스탯 연동 없음.** M1 고정 8개, 순수 연출 | — | **✓** |

- UI 는 `computeStats(02) → ShotProfile(04)` 경로의 출력만 읽는다 — 거짓 프리뷰가 구조적으로 불가능(09 §4 와 동일 원칙).
- **자유 변형의 체감 채널 3종(연속 별 채움·퍼짐 원 프리뷰·사격장 바로가기)은 §5 슬라이더 패널에 배치한다 — 사양 정본은 09 §4.** 06 은 배치·레이아웃만 소유.
- 사격장 HUD 상단에 **현재 총 별 요약 1줄**(파워·정확·연사) 상시 표시 — 공방에서 바꾼 별이 사격장까지 따라온다는 연결 고리.
- 파츠 교체·변형 직후 사격장 진입 시, 변화한 스탯의 별이 HUD 에서 한 번 더 노랗게 반짝(0.4초) — "아까 바꾼 게 이거야".
- 조준 링 스펙: 링 반지름(px) = 04 퍼짐각을 기준 과녁 거리(10m)에서 화면에 투영한 크기. 정확이 낮으면 크고 헐렁한 링, 높으면 작고 탱탱한 링 — 파츠를 끼우거나 배럴을 늘이는 순간 링이 조여지는 애니메이션(0.3초)으로 개선을 즉시 보여준다.

## 8. 사운드 · 주스(juice)

전작 `src/game/audioKit.ts`(절차 합성, 외부 음원 0)를 그대로 가져온다. 핵심 원칙도 승계: *클릭 없는 어택(0.0001→ramp)·필터드 노이즈·배음 적층·전부 작고 둥글게*. **전작 교훈(메모리): 저역·노이즈 위주 소리는 오류음처럼 들린다 → 이 게임 소리는 전부 밝고 높게.**

**부팅 태스크(결정문 26): 첫 사용자 입력(클릭/탭)에서 `AudioContext.resume()`** — 자동재생 정책으로 무음이 되는 사고 방지. M1 필수.

| 이벤트 | 소리 스케치 (audioKit `tone` 파라미터 감각) | 시각 주스 |
|---|---|---|
| 버튼 탭 | 목금 "톡" — sine 880Hz, 0.05s | 버튼 0.1s 눌림 스케일 0.94 |
| 파츠 부착 | **"찰칵"** — square 1200→900Hz 글라이드 0.08s + 클릭 노이즈 한 점 | 파츠 팝(1.0→1.15→1.0, 0.15s) + 슬롯 링 반짝 |
| 색 채우기 | "퐁" — sine 660→990Hz, 0.12s | 색이 존에 0.2s 번져나감 |
| 발사 | **"퓽!"** — triangle 1400→400Hz 하강 글라이드 0.15s (M1 이후: 파워만큼 시작 피치 ↓) | 총구 스프링 반동 + 다트 트레일 |
| 명중 | "팡+짤랑" — 노이즈 버스트 0.05s + sine 1760Hz 0.2s | 풍선 팝 + 색종이 파티클 8개 고정 (§7) |
| 되돌리기 | 되감기 "쀼룩" — 660→880Hz 상승 | 파츠가 역방향으로 스르륵 |

변형 슬라이더 사운드("뽀롱" 피치 슬라이드·딸깍)는 M1.5 — 정본은 09 §5. 발사음은 절차 합성이 우선이되, 나중에 실음원을 얹고 싶으면 전작 `sfxPlayer.ts` 의 **"샘플 재생 실패 시 절차 합성 폴백(무음 없음)"** 구조를 그대로 재사용한다.

## 9. 저장 정책 — 저장 버튼 없음, 전부 자동저장 (결정문 14 — 이 문서가 정본)

아이의 창작물 유실은 이 장르 최악의 사고이고, 수동 저장 버튼은 아이가 안 누른다. **저장은 UI 에서 개념 자체를 소거한다.** 세이브 루트 스키마·SAVE_VERSION·`blaster_lab_` 키 접두사의 정본은 05(결정문 1·13) — 여기는 트리거와 UX 계약만.

- **자동저장 트리거 4종 (즉시 flush, debounce 없음 — 저장물은 수 KB 라 localStorage 동기 쓰기 부담 없음):**
  1. 조립 변경 직후 — 파츠 부착/분리 + **변형 슬라이더 pointerup·프리셋 버튼**(09 §5 가 이 트리거로 flush)
  2. 색 변경 직후
  3. 스테이션 전환 완료 직후 (마지막 스테이션 기억 포함 — §3 재방문 시작점)
  4. `visibilitychange → hidden` / `pagehide` (탭 닫기·홈 버튼 안전망)
- **저장 실패 처리(결정문 26)**: 조용히 즉시 재시도 → 그래도 실패하면 **백업 키에서 복원 + 안내 화면**(그림 중심, 한 줄 문구). 침묵 유실 금지 — 단 매 실패마다 다이얼로그를 띄우지는 않는다.
- 05 에 전달하는 인터페이스 요구:

```ts
// 05 소유 — UX 가 요구하는 최소 계약. Blaster 타입 정본 = 02 (결정문 12, morph 포함)
export interface AutosaveApi {
  flush(blaster: Blaster, lastStation: StationId): void; // 동기. 실패 시 재시도→백업 복원+안내 (결정문 26)
}
```

- **브러시(획) 페인팅 도입 시 `paintOps[]` 배열 추가 + SAVE_VERSION 마이그레이션이 선행 조건** — 이 요구를 05 가 스키마 확장 여지로 미리 반영해 둘 것. morph 는 SAVE_VERSION 1 부터 포함(결정문 13, 직렬화 정본 09 §6).

## 12. 현재 구현 — PVP 보관함 로비와 전투 HUD (2026-07-24)

- 로비는 보관함 작품 수만큼 카드를 만들고 파워·연사·정확·다루기를 별 5개로 보여준다.
- 전투 HUD는 양측 `체력 n / 10`, 라운드 `1 / 3`, 퍼짐 링, 조작 힌트를 표시한다.
- 라운드 결과 오버레이는 다음 라운드·체력 10으로 다시·보관함으로 세 동작만 제공한다.
- PVP는 편집/사격장의 8px 탭·드래그 판별과 달리 마우스의 화면 절대 위치를 목표 조준점으로 사용한다.
- 경기 시작 시 선택 작품을 스냅샷으로 고정해 경기 도중 보관함 변경이 진행 중 라운드를 바꾸지 않는다.
