# 사격 시뮬레이션 · 사격장

> **TL;DR**: 발사체는 **실 투사체(포물선, 오일러 적분) + 선분-구 스윕 판정** — 히트스캔이면 탄속·낙차 스탯이 죽는다. **M1 코스 = "풍선 마당" 1개(고정 과녁 3 + 풍선 6 팝 + 간이 별점[명중 수 기준] + 무한 탄창)**, 정식 점수판·콤보·별 컷·재장전은 M2, 캔(간이 물리)+무빙 M3, 타임어택+슬로모 M4. 조작은 **1인칭 사대 고정 + 드래그 조준 + 탭/클릭 발사**(Pointer Lock 안 씀, 결정문 18). 스탯→발사 변환은 **이 문서의 ShotProfile 계약이 단일 정본**(탄속 20~60m/s·퍼짐 0.3~4.0°·연사 간격 150~600ms — 결정문 4). 콤보는 빗나감 리셋이 아니라 **시간 기반(1.5초 무히트 시 리셋)**, 에임 어시스트는 2층(M1 판정 반경 → M2 탄도 해 자석), 반동은 `복귀율 = max(8, 상승률×0.7)` 불변식. 발사체 크기는 **`projectileScale` 시각·충돌 공용 단일 필드**(09 자유 변형의 bore 기여 수용, §2.3). 밸런스는 시드 고정 몬테카를로 + 상대 불변식으로 verify 게이트에 강제. 성능 예산 수치의 정본 = 07 단일표(결정문 23).

## 1. 코스와 마일스톤 배치 — 모드 = 유형, 코스 = 인스턴스 (결정문 9)

모드 5종은 **유형(enum `RangeModeId`)** 으로 이 문서가 의미·판정 규칙을 소유하고, 코스(스폰 배치·별 컷)는 **05 `CourseDef` 단일 코스 테이블에 병합된 인스턴스**다. 사격 공간의 한글명은 **"사격장"**(06 StationId `range` — 결정문 8).

| 모드(유형) | 내용 | 핵심 검증 스탯 | 마일스톤 | 구현 난도 |
|---|---|---|---|---|
| **고정 과녁+풍선 레인** (`targetLane`) | **M1 코스 인스턴스 = "풍선 마당"**: 거리별 링 과녁 **3개**(8m/15m/25m) + **풍선 6개**(맞으면 팝+콘페티+히트 숫자, 3초 후 재생성). 무한 탄창, 무한 사격. **간이 별점(명중 수 기준) 포함** | 탄속(낙차)·퍼짐 | **M1** (첫 플레이어블, 결정문 20) | 하 — 정적 메시 + 판정 + 메시 숨김/파티클 |
| **풍선 팡팡** (`balloonPop`) | 풍선 12개 벽. 다 터뜨리면 클리어. 정식 점수판·콤보·별 컷 여기서 도입 | 연사·퍼짐 | **M2** | 하 — M1 풍선 재사용 + 라운드 규칙 |
| **캔 쓰러뜨리기** (`canKnock`) | 캔 피라미드(6+5+4=15개). 맞으면 넘어지고 잠깐 구른 뒤 퇴장(간이 물리 §7) | 발사체 종류(무게)·정확도 | **M3** | 중 — 간이 rigidbody 자작 |
| **무빙 타겟** (`movers`) | 레일 왕복 과녁(1~2.5 m/s) + 8자 비행 장난감 드론 | 탄속(리드샷) | **M3** | 중 — sin 기반 모션 |
| **타임어택** (`timeAttack`) | 45초, 풍선 자동 재생성, 콤보 배수 극대화 | 종합(DPS 체감) | **M4** | 하 — M2 재사용 + 타이머 |

**M1 재정의 근거**: "무반응 고정 과녁 무한 사격"만으로는 첫 플레이어블이 가장 지루한 모드가 된다. 풍선 팝은 어차피 난도 '하'(메시 숨김+파티클)고 히트 숫자·팝 사운드는 비용 ~0 — 가장 싼 재미를 M1에 앞당긴다. "맞으면 터지고 숫자가 튄다"가 M1의 최소 재미 보장선이다. 결정문 20 에 따라 **간이 별점(명중 수 기준, 상수 3개 — 제안값 ★1=8 / ★2=15 / ★3=25 히트)은 M1**, 정식 점수판·콤보·별 컷·라운드 규칙(10발 고정 등)·탄창/재장전은 M2 — **M1은 무한 탄창(재장전 없음, 결정문 21)**으로 못박아 조용한 비대화를 차단한다(`magSize`/`reloadMs`는 타입에는 있되 M1에선 미사용, 잔탄 UI는 "∞" 고정). M1 로스터에 탄창 파츠가 없는 것과 짝이다. 또한 필수 슬롯이 body 뿐이므로(결정문 16) **맨몸(body 만) 블래스터로도 풍선 마당 사격이 성립**해야 한다 — 맨몸 기본 발사값은 02 `computeStats` 가 정의하되 이 문서의 계약·클램프를 그대로 통과한다.

**M1 30분 증분 분해**: ① 오일러 탄도 + 스윕 판정 + 과녁 존 판정(점수는 콘솔 로그) → ② 풍선 6개 + 팝 파티클 + 히트 숫자 DOM → ③ 팝/발사 사운드 + 반동 킥 + 판정 반경 어시스트 + 간이 별점 표시.

캔 물리는 물리엔진 도입 없이(§7) 간다 — 전작도 외부 물리엔진 없이 전 콘텐츠를 소화했다.

## 2. 발사체 모델 — 투사체(포물선) 채택 (M1 정본, 결정문 19)

**히트스캔 탈락 근거**: 이 게임의 재미 축은 "파츠를 바꾸면 — 그리고 주무르면(09) — 총알이 *다르게 날아간다*"이다. 히트스캔이면 탄속·낙차·리드샷이 전부 무의미해지고, 스탯 차이가 숫자 팝업 차이로만 남는다. 초등학생에게는 "젤리볼이 무지개처럼 뚝 떨어지는 게 눈에 보이는 것"이 스탯표 10줄보다 강하다.

**구현**: 전작 `C:/ai-game-lab/src/game/guardProjectiles.ts`의 leaf 투사체 모듈 패턴(스탯 `Record` 테이블 + 역순 순회 update + main.ts 미import)을 계승하되, 전작의 "착탄점 lerp + sin 아치"(발사 순간 착탄점 확정) 방식은 버린다 — 플레이어 사격은 착탄점을 미리 모르므로 **오일러 적분 탄도**로 바꾼다:

```
velocity.y -= gravity * dt;  position += velocity * dt;
```

**터널링 방지 필수**: 탄속 60 m/s @ 60fps = 프레임당 1m 이동 > 풍선 반경 0.35m. 점 판정이면 관통한다. **이전 위치→현 위치 선분 vs 타겟 구(sphere) 스윕 판정**을 M1부터 넣는다(판정 구 반경 = 타겟 `hitRadius` × 어시스트 계수 + 발사체 반경 §2.3). 브루트포스 64발×30타겟 = 1,920회/프레임 — 예산 내(공간 분할 불필요).

### 2.1 중력·기본 반경은 발사체 종류의 파생값 — 단일 소스 테이블 (kind 표기 정본, 결정문 6)

`gravity`를 `ShotProfile` 필드로 두지 않는다. `kind`와 `gravity`가 둘 다 있으면 조립 섹션이 임의값을 채워 "물감볼인데 다트처럼 나는" 비정합이 생기고, 파생값 중복 저장은 세이브 마이그레이션의 desync 소스다. **shooting leaf가 소유하는 상수 테이블로 단일화** — 조립 섹션은 `kind`만 넘긴다. kind 표기는 이 표가 정본이다(결정문 6 — `suctionArrow` 류는 존재하지 않는다).

| kind | gravity (m/s²) | 기본 반경 (m) | 특성 | 도입 |
|---|---|---|---|---|
| `dart` (다트) | 4 | 0.025 | 기준 발사체. 직선에 가까움 | M1 |
| `gel` (젤리볼) | 6 | 0.035 | 완만한 무지개 궤적 | M1 |
| `paint` (물감볼) | 14 | 0.045 | 낙차 큼 — 위로 조준해야 함 | M2 |


### 2.2 파츠 스탯 → 시뮬 변수 → 체감 매핑 표 — 스탯→발사 변환의 단일 정본 (결정문 4)

**이 표와 클램프가 스탯→발사 계약의 정본이다** — 02 가 6스탯(1~10)에서 이 범위로의 매핑을 신설·재보정하고, 06 §7 의 독자 수치 테이블은 폐기되어 이 표를 참조한다. (스탯이 파츠 합산 + morph 델타로 *어떻게 산출되는지*는 02 담당 — §10.)

| 스탯 | 시뮬 변수 | 범위 | 눈·손 체감 |
|---|---|---|---|
| 파워 | `muzzleVelocity` | **dart 20~60 / gel ≤54 / paint ≤42 m/s** | 종류별 상한은 `PROJECTILE_DEFS.maxSpeed`에서 파생 |
| 정확(퍼짐) | `spreadDeg` (원뿔 반각) | **0.3~4.0°** | 15m에서 착탄 산포 반경 8cm~1.0m. 과녁 정중앙이 되냐 안 되냐 |
| 낙차 | `PROJECTILE_GRAVITY[kind]` | §2.1 테이블 | 궤적 곡선이 눈에 보임 |
| 반동 (파생) | `recoilKickDeg` | 0.2~2.5°/발 (파생 규칙은 02, 클램프는 §3) | 카메라 pitch 킥. 체감 3구간은 §3 |
| 연사 | `fireIntervalMs` | **150~600ms (400~100 RPM)** | 손맛 리듬. 트리거 홀드 시 발사음 밀도 |
| 다트 수(capacity)·재장전 | `magSize` 6~30 / `reloadMs` 800~2500 | M2부터 활성 (결정문 21·25 — "체감 없는 스탯 금지") | "쏘고-갈아끼우는" 호흡. 타임어택 전략 변수 |

### 2.3 발사체 크기 — `projectileScale` 시각·충돌 공용 (09 요청 수용, 신설)

09 자유 변형의 배럴 굵기(bore) 기여를 받아 **발사체 크기를 시각·충돌 공용 단일 필드로 정의한다** — 시각 전용 크기 채널은 두지 않는다(보이는 대로 맞는다). 합성식·충돌 반경 동기화의 정본은 이 절이다:

```ts
// shooting leaf 소유 순수 함수 — 02 의 ShotProfile 산출기가 호출
// power: 최종 파워 스탯 1~10, boreScale: 09 morph 기여(제안 lerp(0.9, 1.3, t), morph 없으면 1.0)
export function composeProjectileScale(power: number, boreScale: number): number {
  return clamp(boreScale * (1 + 0.02 * (power - 5.5)), 0.85, 1.35);
}
```

- 시각 메시 스케일 = 충돌 반경 스케일 = `projectileScale` 하나. 실 반경 = `kind 기본 반경(§2.1) × projectileScale` — 스윕 판정(§2)과 렌더가 같은 값을 읽는다.
- morph 미도입 파츠·맨몸은 `boreScale = 1.0` — 파워 채널만 남아 기존 동작과 연속.

## 3. 반동 모델 — 퇴화 방지 불변식

복귀율 8°/s 고정은 퇴화한다: 예시 조합 A(120RPM×1.2°)는 상승 2.4°/s, B(400RPM×0.35°)는 2.3°/s — 둘 다 복귀율 미만이라 "기어오름"이 어디서도 안 일어나고, 반대로 극단 조합(400RPM×2.5°=16.7°/s)이 무가드로 허용되면 제어감이 무너진다. 아래 3규칙으로 고친다.

1. **복귀율 동적화**: `recoveryDegPerSec = max(8, riseRate × 0.7)`, 단 `riseRate = recoilKickDeg × (1000 / fireIntervalMs)`. 어떤 조합이든 순상승률이 `min(riseRate − 8, 0.3 × riseRate)` 이하로 눌려 "천천히 기어오르되 제어 가능" 범위에 남는다.
2. **누적 상한**: `RECOIL_MAX_DEG = 6` — 아무리 갈겨도 하늘을 보게 되진 않는다.
3. **조립 섹션 전달 클램프 (계약)**: `recoilKickDeg × (1000/fireIntervalMs) ≤ 20 (°/s)` — 연사 간격 하한 150ms(결정문 4) 기준 재보정값. 이 불변식은 §4 몬테카를로 테스트가 유효 스탯 범위 샘플링으로 검증한다.

체감 3구간 설계 (기본 파츠 범위는 1구간에 두고, 고반동 파츠 — 강화 스프링류 — 가 2구간을 연다):

| 구간 | 조건 | 체감 |
|---|---|---|
| 킥-복귀 | riseRate ≤ 8°/s | 발마다 화면이 툭 들렸다 돌아옴 — 단발 손맛. 예시 조합 A·B 모두 여기 |
| 기어오름 | riseRate > 8°/s | 예: 400RPM×1.5° = 10°/s → 순상승 초당 2° — 끊어 쏘기 요령이 생김 |
| 상한 | 누적 6° 도달 | 더 안 오름. 연사를 멈추면 복귀 |

## 4. 스탯→체감 검증 — 같은 과녁, 다른 조합

**검증 시나리오**: 15m 링 과녁(반경 0.5m, 중앙 존 0.15m)을 두 조합으로 사격. (조합명은 명사수/정밀 계열 — 결정문 25, "스나이퍼/저격" 금칙.)

| | 조합 A "명사수 다트" (긴 배럴+정밀 스코프) | 조합 B "젤리 분무기" (짧은 배럴+대용량 드럼) |
|---|---|---|
| 탄속 / 낙차 | 50 m/s, 다트 g=4 → 도달 0.30s, **낙차 18cm** | 28 m/s, 젤리볼 g=6 → 도달 0.54s, **낙차 87cm** |
| 퍼짐 | 0.5° → 산포 반경 **13cm** (거의 조준점 그대로) | 3.0° → 산포 반경 **79cm** (과녁 밖까지 튐) |
| 연사 / 탄창 | 120 RPM, 6발 | 400 RPM, 30발 |
| 반동 | 상승 2.4°/s — 발마다 툭 들렸다 완전 복귀 (킥-복귀 구간) | 상승 2.3°/s — 잔잔한 떨림 (킥-복귀 구간) |
| **아이 눈에 보이는 차이** | 쏘면 거의 직선, 조준한 곳에 팍 박힘 | 무지개 궤적으로 우수수 떨어지고 사방에 튐 |
| 예상 명중률(과녁 내) | **~90%** | **~40%** (대신 초당 6.7발이라 총 히트 수는 비슷) |

이 대비가 설계의 심장이다: **"정확한 한 발" vs "뿌리는 물량"이 서로 다른 방식으로 같은 점수에 도달**해야 어느 조합도 오답이 아니게 된다. (예시 조합의 스코프·드럼은 M2 로스터 — M1 검증은 로스터 5종+맨몸(결정문 21) 범위에서 같은 구도로 축소 실행한다.)

### 자동 검증 — 시드 고정 + 상대 불변식 (verify 게이트)

절대값 어서션(`hitRateA > 0.85` 등)은 넣지 않는다 — 아이 피드백으로 스탯을 계속 만질 프로젝트라 spreadDeg 하나 고칠 때마다 게이트가 깨지고, 시드 없는 몬테카를로는 경계값 근처에서 flaky다. 게이트에는 **튜닝해도 참이어야 하는 상대 불변식**만 둔다:

```ts
const rand = mulberry32(0xC0FFEE);            // 시드 주입 — 결정론적, flaky 차단
const A = simulateRound(PRECISION_DART, RANGE_15M, 1000, rand);   // 결정문 25: SNIPER_DART → PRECISION_DART
const B = simulateRound(GEL_SPRAYER, RANGE_15M, 1000, rand);

expect(A.hitRate).toBeGreaterThan(B.hitRate + 0.2);   // 정밀 빌드가 확실히 더 잘 맞는다
expect(A.meanDropM).toBeLessThan(B.meanDropM);        // 빠른 다트가 덜 떨어진다
const ratio = simulateTimeAttack(A, 45_000, rand).score
            / simulateTimeAttack(B, 45_000, rand).score;
expect(ratio).toBeGreaterThan(0.8);                   // 두 빌드 최종 점수 균형 —
expect(ratio).toBeLessThan(1.25);                     // "설계의 심장"을 수치로 강제
// 반동 불변식 (§3-3): 유효 스탯 범위 500조합 샘플링 — morph 델타 반영 조합 포함(§10)
expect(maxRiseRate(sampledProfiles)).toBeLessThanOrEqual(20);
```

절대값 리포트(명중률 실측표·낙차표)는 게이트가 아니라 수동 실행 스크립트 `npm run report:balance`로 분리 — 튜닝할 때 참고용으로 찍어본다. 실화면은 전작 playwright 헤드리스 스크린샷 하네스로 궤적 곡선을 촬영해 눈으로 최종 확인.

- 히트마다 `comboCount + 1`. **마지막 히트로부터 1.5초간 히트가 없으면 리셋.** 빗나감 자체는 리셋 조건이 아니다.

- 수치 확인: 연사 빌드 400RPM×40% ≈ 초당 2.7히트(히트 간격 ~0.38s), 단발 빌드 120RPM×90%(~0.55s) — 양쪽 다 조준을 유지하는 한 콤보가 이어지고, 리셋은 "조준을 놓치고 헤매는 구간"에서만 발생한다.

## 7. 캔 "간이 물리" — 물리엔진 없이, 관통 노출 없이

cannon-es 등 도입 금지(번들·전작 무경험). 캔-캔 충돌은 **구현하지 않고, 관통이 보일 시간을 없앤다** — 넘어진 캔이 옆 캔을 뚫고 겹쳐 머무는 글리치는 초등학생도 즉시 알아채므로, 겹침의 화면 체류를 1초 미만으로 끊는다. 상태는 2개뿐:

1. **standing**: 정적. 피격 시 → falling. `velocity = 발사체방향 × (탄속/20) + up 1.2`, `angularVelocity` 랜덤 4~8 rad/s. 위층 캔은 아래층 지지 캔 파괴 시 0.15초 지연 후 연쇄 falling(지지 관계는 스폰 시 배열로 고정 — 그래프 탐색 불필요, STRIKE 연쇄 판정도 이 배열로).
2. **falling**: 중력 9.8 적분 + 회전, 바닥 접촉 시 바운스 1회(반발 0.3) → **0.5초 구른 뒤 스케일 축소 페이드아웃(0.4초) + 점수 확정 + 풀 반환**.

`settled` 상태·3초 타이머는 통째로 삭제 — 상태머신이 오히려 단순해진다. 캔 15개 전부 falling이어도 15회 적분/프레임 — 무시 가능한 비용.

## 8. 시점 · 조작 · 에임 어시스트 (결정문 18·22·24)

**1인칭 + 사대 고정 + 드래그 조준 + 탭/클릭 발사 채택 (결정문 18). Pointer Lock 은 쓰지 않는다.** 조준은 화면 드래그로 look 을 돌리고, 발사는 탭/클릭 — **탭 vs 드래그 판별은 8px 이동 임계**로 pointer 이벤트 레이어에서 처리한다(결정문 24 의 공방 판별 규칙과 동일 상수·동일 레이어 공유). 트리거 홀드(연사)는 판별 확정 후의 press 유지로 정의한다.

근거: (1) 화면 하단 뷰모델이 **내가 주무르고(09) 색칠한 총을 상시 노출** — 자유 변형·꾸미기 기둥과 시뮬레이션 기둥이 한 화면에서 만나는 지점이라 양보 불가. 뷰모델은 09 §3.1 의 동일 `buildPart` 빌더를 공유하므로 "공방에서 늘인 총이 사격장에서 원래대로" 같은 배신이 구조적으로 불가능하다. (2) 3인칭은 캐릭터 모델(없음)과 조준 오프셋 보정이 필요해 M1 예산 초과. (3) 사대 고정(이동 없음)이라 초등학생 조작 부담이 WASD+마우스 동시 조작 대비 절반. 레인 이동은 M2에서 "사대 A/B/C 버튼 순간이동"으로 — 자유 보행은 넣지 않는다. **플랫폼은 PC 데스크톱 우선(결정문 22) — 터치 대응은 M2**이며, 그때 전작 `src/ui/touchControls.ts` leaf 패턴을 계승한다(상세는 UI/플랫폼 섹션).

**에임 어시스트 — 2층 구조 (기본 ON. OFF 토글 = 결정문 26 의 설정 최소 토글 2종 중 "에임 어시스트" — "고수 모드")**:

| 층 | 내용 | 데스크톱 (M1 기준) | 터치 (M2) | 도입 |
|---|---|---|---|---|
| ① 판정 뻥튀기 | 판정 반경 = 시각 반경 × 계수 (상수 1줄) | ×1.25 | ×1.4 | **M1** |
| ② 자석 스냅 | 조준점 근처 타겟에 대해 **탄도 해 기반 발사각 보정** | 탐지 3.0° / 최대 보정 1.5° | 탐지 4.5° / 최대 보정 2.5° | M2 |
| 궤적 점선 가이드 | 예상 포물선 12개 도트 (InstancedMesh 1개, 10Hz 갱신) | 어시스트 ON이면 **항상 표시** | 동일 | M2 |

**자석 스냅은 방향 보정이 아니라 탄도 해 보정이다.** 타겟 *방향*으로 1.5° 틀어봤자 낙차 87cm짜리 발사체는 여전히 타겟 아래에 착탄한다 — 보정 목표각은 포물선 조준각의 닫힌 해(순수 함수, 단위테스트 대상)로 구한다:

```ts
// 수평거리 d, 높이차 h, 탄속 v, 중력 g → 저각 해 pitch (사거리 밖이면 null → 개입 안 함)
export function solveBallisticPitchRad(d: number, h: number, v: number, g: number): number | null {
  const disc = v ** 4 - g * (g * d * d + 2 * h * v * v);
  if (disc < 0) return null;
  return Math.atan((v * v - Math.sqrt(disc)) / (g * d));
}
```

보정은 이 해 방향으로 발사 각도를 최대 1.5°/2.5°만 트는 방식이라 강제 명중이 아니고, **타겟의 현재 위치 기준으로만 계산**하므로 무빙 타겟 리드샷 학습은 그대로 남는다. 궤적 가이드를 튜토리얼 전용으로 숨기지 않는 이유: 낙차가 이 게임의 핵심 학습인데 가이드가 곧 교재다 — 고수 모드 OFF가 곧 가이드 OFF.

## 10. 자유 변형(09) 과의 결합 — 시뮬레이션 관점 (정본은 09, 결정문 27)

자유 변형의 상세 사양(파라미터·지오메트리·UX·직렬화)은 **09 가 소유한다**. 이 문서가 지는 책임은 아래 4개 접점뿐이다:

1. **스탯 경로 불변**: morph 스탯 델타는 02 `computeStats`(morph 인지형)가 합산하고, 이 문서는 결과 `ShotProfile` 만 받는다 — §2.2 클램프가 최종 방어선이라 **morph 가 발사 계약을 뚫을 수 없다**.
2. **`projectileScale` 소유**: 09 §2.2 의 bore 기여(`boreScale`)를 받아 파워 채널과 합성하는 식·충돌 반경 동기화는 §2.3 이 정본 — 09 의 개정 요청을 수용해 신설했다.
3. **매핑 함수 export**: 스탯→`ShotProfile` 변환은 순수 함수로 export 한다 — 09 §4 의 "퍼짐 원 프리뷰"가 동일 경로를 호출해 **거짓 프리뷰가 불가능**해지는 근거.
4. **몬테카를로 커버리지**: §4 반동·상대 불변식 샘플링에 morph 델타 반영 조합을 포함한다(극단 morph 로도 §3-3 클램프 유지 검증).

## 13. PVP AI 드론 팝 아레나 (2026-07-24 현재 구현)

- **라운드**: 무표정 구형 AI 드론 3기와 순차 1대1. 양측 체력은 매 라운드 10이며, 드론 완료 시 다음 라운드, 재도전·무승부는 같은 라운드를 다시 시작한다.
- **팝 파워**: `popPower = 0.75 + (power - 1) / 12`로 0.75~1.5. 체력은 0 아래로 내려가지 않는다.
- **연사·정확**: 기존 `ShotProfile`의 `fireIntervalMs` 600~150ms와 `spreadDeg` 4.0~0.3°를 그대로 사용한다.
- **다루기**: `t = (handling - 1) / 9`, `aimFollowPerSec = 6 + 12t`, `strafeSpeed = 1.8 + 1.8t`. 플레이어 조준 추종·좌우 피하기와 AI 추적에 같은 축을 사용한다.
- **물리**: 발사체 종류·탄속·중력·크기와 스윕 구 충돌은 사격장 정본을 재사용한다. 모드별 64발 풀, 고정 시드 `XorShift32`, AI 첫 반응 700ms를 사용한다.
- **공정성**: 같은 프레임 양측 명중은 이전 체력 기준으로 동시에 계산한다. `capacity`·`reload`는 PVP 1차 범위에서 제외한다.

## 조립 섹션(02)에 넘기는 계약 (요약 — 이 계약이 스탯→발사 정본, 결정문 4)

1. `ShotProfile`은 `kind`만 넘긴다 — `gravity` 필드 없음 (§2.1).
2. 클램프 4종: `muzzleVelocity` 20~60 + 종류별 maxSpeed, `spreadDeg` 0.3~4.0, `fireIntervalMs` 150~600, `recoilKickDeg × (1000/fireIntervalMs) ≤ 20` (§3).
3. `projectileScale` 은 이 문서 소유 순수 함수 `composeProjectileScale(power, boreScale)` 로 산출 (§2.3) — 02 는 파워 스탯을, 09 morph 는 `boreScale` 기여를 입력으로 넘긴다. 시각 전용 크기 채널 금지.
4. `ShotProfile` 산출 공식(가중 합산 vs 곱연산)은 02 가 결정하되 §4 상대 불변식 테스트를 통과해야 한다. **맨몸(body 만) 기본 발사값도 02 `computeStats` 가 정의**(결정문 16)하며 같은 클램프를 지난다.

## 기각한 비평

- **비평 4의 진단 중 "조합 A는 반동 체감 0" 부분만 기각**: 발당 1.2° 킥이 튀었다 복귀하는 것 자체가 의도된 단발 손맛이며(§3 "킥-복귀 구간"), 체감 0이 아니다. 다만 같은 비평의 핵심 — "기어오름이 어떤 예시 조합에서도 발생하지 않고, 극단 조합 폭주를 막는 가드가 없다" — 은 타당하므로 수정안(동적 복귀율 + 불변식 클램프)은 전면 채택했다.
- 나머지 비평 7건(1·2·3·5·6·7·8)은 전부 타당해 반영: M1 풍선 팝 포함, 시간 기반 콤보, 2층 어시스트 + 탄도 해, 시드 고정 + 상대 불변식 게이트, gravity·반경·속도·시각 정책의 `PROJECTILE_DEFS` 단일화, 캔 페이드아웃 퇴장.
