# 진행 · 해금 · 수집 · 저장

> 전제: 파츠 로스터·스탯 축의 정본은 **02 조립 섹션**(결정 7), 코스 구성·점수 산식·발사 계약의 정본은 **04 시뮬레이션 섹션**, 자유 변형(파라메트릭 조형)의 정본은 **09 자유 변형 섹션**(결정 27)이다. 본 섹션은 그 위에서 "왜 내일 또 접속하는가"를 만든다. 설계 원칙 한 줄 — **다음 목표가 항상 화면에 보인다** (잠긴 파츠는 숨기지 않고 회색 카드+해금 조건으로 노출).
>
> 경제는 시뮬레이션 섹션의 점수 스케일에 **결합하지 않는다**. 코인 보상은 코스 데이터 테이블의 명시 컬럼(`baseReward`·`starBonus`)에서만 나오고, 점수는 별(★) 판정 임계값으로만 소비된다. 시뮬 섹션이 점수 산식을 바꿔도 경제는 무풍.
>
> **결정문 반영 핵심 2줄**: ① 코인 경제 전체(상점·정산·가격표)는 **M2 로 이동**(결정 5) — M1 의 진행 축은 **★ 직결 파츠 해금 + 자유 변형(09)** 이다. ② 조립 데이터는 단일 타입 `Blaster`(결정 12)로, **morph 포함 SAVE_VERSION 1 부터** 저장한다(결정 13).

## 1. 보상 통화 — 스파크 코인 1종 (도입 = M2, 결정 5)

통화는 **1종만** 둔다. 초등학생에게 이중 통화(코인+젬)는 혼란만 주고, 수급 밸런스 관리 비용이 2배가 된다. 별(★)은 통화가 아니라 **코스 평가 등급**(0~3개)으로만 쓴다. **M1 은 코인이 아예 없다** — `coins` 필드는 v1 스키마부터 존재하되 항상 0 이며(마이그레이션 회피), UI 에 코인 표기가 등장하는 시점이 M2 다.

| 항목 | 값 |
|---|---|
| 이름 | 스파크 코인 (UI 표기: 절차 SVG 코인 아이콘 + 숫자 — 이모지 금지, 결정 25) |
| 용도 | 파츠 구매, 색상/패턴/스티커 구매, 실력 게이트 코인 우회(§2), (M3) 진열장 확장 |
| 저장 필드 | `SavedGame.coins: number` |

### 획득 규칙 — 전부 코스 데이터 테이블의 명시 컬럼에서 (M2 부터 활성)

코스는 **04 모드 5종과 병합된 단일 코스 테이블**이다(결정 9 — 모드=유형, 코스=인스턴스). 코스 구성·별 판정 산식은 04 정본, 이 섹션은 **보상 컬럼만** 소유한다.

```ts
// src/game/courses.ts — 단일 코스 테이블 (결정 9). 점수→코인 변환 없음
interface CourseDef {
  id: string;
  mode: CourseMode;              // 04 모드 5종 중 하나 (유형) — 정본 04
  baseReward: number;            // 완주 반복 보상의 기준값 (M2 활성)
  starBonus: [number, number, number]; // ★1/★2/★3 "첫 달성" 보너스 (누적 지급)
  // ...코스 구성·별 임계는 04 정본
}
```

| 획득처 | 코인 | 비고 |
|---|---|---|
| 코스 완주 (반복 가능) | `round(baseReward × (0.75 + 0.25 × 이번 판 별))` | ★1=1.0배, ★2=1.25배, ★3=1.5배 — **잘 쏘면 반복 보상도 는다** |
| 오늘 그 코스 첫 플레이 | 위 완주 보상 **2배** | 로컬 날짜(YYYY-MM-DD) 기준, `dailyBonus` 필드. M2 도입 |
| 코스×별 **첫 달성** 보너스 | `starBonus[s]` 를 새로 오른 별마다 누적 지급 | ★1 은 완주 시 확정 → **첫 클리어 보너스 = ★1 첫 달성 보너스** (별도 항목 없음) |
| 도전 과제 달성 | 50~300 | §4 표 참조. 1회성 지급은 전부 과제로 일원화. M1 과제 2종의 코인 보상은 **M2 코인 도입 시 소급 지급**(achievements 장부가 이미 있으므로 코드 1줄) |

**수급 속도 견적 (M2 기준)**: 코스 1회 ≈ 2~3분. 중반 코스(base 50~60)를 ★2~★3 으로 반복하면 시간당 **약 1,400~1,800코인**. 세션 첫머리는 일일 2배 보너스로 코스당 +50~90 추가. → **최고가 파츠(T4 850) ≈ 세션 1개(30~40분)** 안에 도달. "파츠 1개에 순수 반복 세션 2개" 그라인드를 만들지 않는다.

## 2. 파츠 해금 트리

### M1 — ★ 직결 해금 (코인 없음, 결정 5·21)

M1 로스터 = **5종 + 맨몸**(결정 21): 몸통 2(팝콘·불도그) + 배럴 2(숏 스냅·롱 레일) + 사이트 1(도트). 파츠 표시명·스탯 델타 정본 = **02 §3 표**(결정 7 — 아들 네이밍 세션에서 교체 가능), 파츠 id 규약 = 03(`body_*` 등). 다트 팩(탄창)은 **M2 부터**(결정 21 — M1 무한 탄창, capacity·reload 스탯도 M2 노출).

| 구분 | 파츠 | 해금 |
|---|---|---|
| 시작 (맨몸 첫 사격 직후) | 팝콘 몸통 · 숏 스냅 배럴 | 첫 코스 완주 시 자동 개방 (온보딩 §7 비트 2) |
| ★ 직결 잠금 | 불도그 몸통 · 롱 레일 배럴 | 풍선 마당 **★2** (회색 카드 + "★2 달성" 조건 노출) |
| ★ 직결 잠금 | 도트 사이트 | 풍선 마당 **★3** |

파츠→★ 매핑은 초안 — 아빠 사전 테스트에서 조정. M1 잠금은 코인 우회가 없으므로 별 기준을 **관대하게**(★2 = 보통 플레이) 잡아 좌절 벽을 만들지 않는다. **핵심**: 파츠가 5종뿐이어도 각 파츠를 슬라이더로 주무르는 **자유 변형(09)이 M1 조합 공간의 본체**다 — "같은 파츠 5종으로도 morph 가 다르면 다른 총"(09 §4). 해금 트리는 그 위에 얹히는 이산 목표다.

| 티어 | 파츠 수 | 개당 가격 | 해금 조건 | 새로 열리는 재미 축 | 배치 |
|---|---|---|---|---|---|
| T1 (기본) | 6 | 200~350 | 코인만 | 스탯 편차 체감 (가벼운 몸통 vs 큰 다트 팩) | **M2** (결정 5 — 상점과 함께) |
| T2 (특성) | 8 | 300~500 | 코인 + 코스2 클리어 | 새 탄종 — 젤리볼(gel)→물감탄(paint)→거품탄(foam), kind 표기 정본 04(결정 6). 연사 배럴 | M2.5 |

**실력 게이트는 벽이 아니라 지름길**: T3/T4 의 실력 조건(지정 과제·★3 N개)에는 **코인 우회(정가 2배)가 병존**한다. 조준 실력이 낮아도 시간(코인)으로 반드시 열 수 있다 — "보이게 잠근" 실루엣이 영구 좌절 벽이 되지 않게. 실력 조건을 채우면 반값에 사는 구조라, 잘 쏘는 것이 여전히 이득이다. 별 판정은 **★1 = 완주만으로 확정**(M2 정식 점수판 기준 — M1 은 명중 수 간이 별점, 결정 20), ★2/★3 임계는 아빠 사전 테스트에서 아이 기준 "★2 = 보통 플레이, ★3 = 잘 친 판 상위 30%" 정도로 관대하게 잡는다.

### 콘텐츠 소진 견적

- 파츠+꾸미기 총 가격 합: **약 15,000코인** (T1 1,700 + T2 3,200 + T3 4,800 + T4 3,200 + 꾸미기 2,200)
- 수급 시간당 1,400~1,800 → 순수 전 해금 **9~11시간**. 별 첫 달성·과제 보상(합 ~2,600)이 이를 약간 단축
- 실력 게이트·코스 진행이 병목이라 체감은 그보다 김. 초등학생 세션 30~60분 × 주 4~5회 기준 **약 2~4주 분량**
- M1 은 5종+맨몸 (해금 목표 2개) — 이산 콘텐츠는 작지만 **자유 변형(09)의 연속 조합 공간이 M1 체류 시간의 본체**. 이후 티어는 데이터 행 추가 + 메시 팩토리 함수로 확장

## 3. 내 컬렉션 — 진열장 (결정 25: "무기고" 명칭 폐기)

- **저장 슬롯**: 데이터는 M1부터 **8슬롯**, M3 = 16슬롯 확장(코인 구매). 슬롯 가득 상태에서 새 슬롯 생성 → 교체할 블래스터 선택 모달.
- **이름 짓기**: 최대 12자, 기본값 자동 생성 `"젤리 스트라이커 #3"` (몸통 이름 + 순번). **자동저장 체계(결정 14)라 "저장 버튼"이 없으므로** 이름 입력 모달은 **새 슬롯 생성 시점**에 뜬다. 이후 개명은 카드에서.
- **UI 단계화**: M1 = 텍스트 카드 목록(선택 시 조립대 뷰에서 라이브 재렌더 — 별도 렌더 경로 없음). M2 = 카드 그리드 + 3D 턴테이블 뷰.
- **썸네일은 세이브에 넣지 않는다** — 에셋이 전부 `Blaster`→절차 생성이므로 이미지는 파생물이다. M2 진열장 진입 시 공유 WebGLRenderer 1개로 256×256 렌더타겟에 블래스터별 순차 렌더(16장 × 단순 프리미티브 = 1프레임 내) → `Map<blasterId, dataURL>` **메모리 캐시**. 캐시 키는 파츠 조합+**양자화 morph(소수 2자리)** 포함(09 M2) — 슬라이더로 주무른 결과가 썸네일에 그대로 반영돼야 "내가 만든 총"이 성립. 수정 시 해당 키만 무효화. 세이브는 순수 데이터만 남아 ≤ 20KB.
- **대표 블래스터**: 1개 지정(`activeBlasterId`) — 사격장(결정 8) 입장 시 기본 장착.
- **카드 스탯 표기**: 스탯 요약은 **별 표기**(6스탯 정본 = 02, 별 5개 UI = 결정 3 — 스탯 숫자 노출 금지). 단 `bestScores`(코스별 최고점)는 **기록값이라 수치 허용**(결정 3, 08 §6 범위) — **가장 최근 플레이한 코스의 베스트** 1줄만 표기해, 쉬운 코스1 고득점이 어려운 코스5 기록을 가리는 일이 없다. 덤: `courseRecords.bestBlasterId` 로 코스 화면에 **"이 코스 1등 발명품"** 뱃지가 공짜로 생긴다 — "내가 만든 총의 성적표"가 창작→검증→개선 루프를 진열장과 코스 화면 양쪽으로 끌고 온다.

### localStorage 키 — M1부터 프로필 구조 (M2 "아빠 vs 나"의 비용을 0으로)

| 키 | 내용 |
|---|---|

//   Blaster { id; name; createdAt; parts: Partial<Record<SlotType, PartInstance>> }  // body 만 필수 (결정 16)

  updatedAt: number;

### morph 직렬화 — 정본은 09 §6

- morph 는 **0..1 정규값 희소 Record** — 기본값(0.5) 키는 저장 생략. 파라미터 범위(min/max) 재보정에도 **마이그레이션 불필요**(정규화 저장의 존재 이유). 모르는 MorphKey 는 조용히 무시(하위 호환). 상세 로드 규칙 전부 09 §6 소유.
- roundtrip 테스트에 **"morph 포함 저장→로드→deepEqual" + "미지 MorphKey 세이브 로드"** 케이스를 M1 부터 포함(09 §6 합의).

### 보상 정산 — 이중 장부 금지 (M2 활성)

`claimedFirstClears` 같은 지급 기록을 따로 두지 않는다. **진실은 `courseRecords` 하나** — 첫 클리어 = 레코드 부재, 별 보너스 = 별 상승분 차액 지급. 지급됐는데 기록 안 됨 / 기록됐는데 미지급 desync 가 구조적으로 불가능. M1 도 `courseRecords.stars` 는 갱신한다(★ 직결 해금 + M2 소급 정산의 근거) — 코인 지급부만 M2 에 붙는다.

```ts
// src/game/economy.ts — 코스 종료 정산 (순수 함수, M2 활성)
export function settleCourse(def: CourseDef, run: CourseRun, save: SavedGame): number {
  const prev = save.courseRecords[def.id];            // undefined = 첫 클리어
  const oldStars = prev?.stars ?? 0;
  const newStars = Math.max(oldStars, run.stars) as 0 | 1 | 2 | 3;
  const firstToday = !save.dailyBonus.coursesPlayed.includes(def.id);
  const repeatMul = 0.75 + 0.25 * run.stars;          // ★1=1.0 ★2=1.25 ★3=1.5
  let payout = Math.round(def.baseReward * repeatMul) * (firstToday ? 2 : 1);
  for (let s = oldStars; s < newStars; s++) payout += def.starBonus[s]; // 새로 오른 별만
  return payout;                                      // courseRecords 갱신은 호출측에서 함께 커밋
}
```

  };

## 7. 첫 10분 온보딩 — 3비트: 쏘기 → 만들기 → 해금 (M1, 코인 없음)

M1 하드 룰 + 결정 5(코인 M2)·16(맨몸 첫 사격)·20(코스 = 풍선 마당 1개)에 맞춰 스크립트를 재구성. 1회성 지급은 전부 도전과제(§4)로 일원화 — 온보딩 전용 지급 없음.

| 비트 | 시각 | 이벤트 |
|---|---|---|
| **1. 쏘기 (맨몸)** | 0:00~2:00 | 접속 → 메뉴 없이 바로 **사격장 "풍선 마당"** — body 만 있는 맨몸 블래스터(결정 16, 기본 발사값 = 02 `computeStats`)로 첫 발사. 간이 별점(명중 수)으로 ★ 개념 체험 |
| **2. 만들기** | 2:00~6:30 | "공방에서 멋지게 만들자" → 시작 파츠 2종 개방 → 스냅 조립 + **슬라이더 변형(09 — 늘이고 부풀리는 첫 경험, 퍼짐 원 프리뷰로 성능 변화 체감)** → 새 슬롯 생성 = `first_build`, 색 1회 = `first_paint` |
| **3. 다시 쏘기 → 해금** | 6:30~10:00 | "쏴보러 가기"(09 바로가기 버튼) → 내 총으로 재도전 → **★2 달성 → 회색 카드였던 불도그·롱 레일이 그 자리에서 열린다**(§2). ★3 조건의 도트 사이트는 회색+조건 표시로 잔존 |

- **해금 검증**: 10분 종료 시점에 ★3 파츠(도트 사이트) 1개가 반드시 남는다 → "조금만 더 잘 쏘면 하나 더"가 정확히 성립. 즉시 해금 2개 + 남은 목표 1개 + 무한한 변형 조합 = 재접속 동기.
- 코스가 1개뿐이므로(결정 20) 코스 잠금 연출은 M2 로. 과제 패널 첫 오픈 연출도 M2(과제가 7종이 된 시점)로.
- **M2 확장**: 코인 도입과 함께 비트 4 "사기"(상점에서 T1 회색 카드+가격, 첫 구매 유도)가 붙는다 — 기존 초안의 "장부 검증"(첫 세션 잔고로 두 번째 파츠를 못 사게 가격 설계) 원칙은 M2 온보딩에 그대로 승계.

**핵심 규칙**: 잠긴 것(파츠·코스·과제)은 전부 **보이게** 잠근다. 10분 종료 시점 플레이어의 머릿속 상태 = "저 도트 사이트 열려면 ★3, 그리고 몸통을 더 길게 뽑으면 어떻게 되지?" — 이것이 재접속 동기의 전부다.

## 열린 질문

1. **M1 ★ 직결 해금 매핑** — §2 초안(★2 = 몸통+배럴, ★3 = 사이트)의 별 배치가 아이 실력 대비 적절한가? 간이 별점(명중 수) 임계와 함께 아빠 사전 테스트로 조정. 코인 우회가 없는 구간이라 ★2 는 "보통 플레이"로 관대하게.
2. **실력 게이트 코인 우회 배율 (M3+)** — T3/T4의 "지정 과제 또는 정가 2배"에서 2배(예: 700→1,400, 순수 반복 약 1시간)가 아이 성향에 맞는가? 좌절 회피형이면 1.5배로 낮추는 선택지가 있음. 아이를 아는 아빠의 판단 필요.
3. **일일 첫 플레이 2배 보너스 유지 여부 (M2)** — 세션 페이싱에는 좋지만 "매일 접속해야 손해 안 봄" 압박이 초등학생 습관에 괜찮은지는 부모 판단 영역. 빼도 경제는 성립(수급이 ~15% 감소).

## 기각한 비평

> 이후 오케스트레이터 결정문(특히 5·14·20·21)이 아래 판단 일부를 덮어썼다 — 충돌 시 결정문이 정본.

- **비평 2의 M1 과제 세트(4~5종) 구성 — 부분 기각.** "M1 은 기본 이벤트만으로 판정 가능한 과제로 재구성"이라는 원리는 수용했으나, 제안된 4~5종은 비평 1의 M1 축소(과제 최소화)와 충돌한다. 비평 1의 범위 규율을 우선해 `balloon_streak_10`·`no_scope`·`collection_5`·`full_dress` 는 M2로 이월했다. 원리는 §4의 "필요 이벤트" 컬럼과 동승 원칙으로 반영됨.
- **비평 1의 "M1 과제 = `first_build` 1개" — 부분 기각.** 비평 6이 요구한 "1회성 지급의 도전과제 일원화"를 만족하려면 온보딩 색칠 보상의 수령처가 필요하다. `first_paint` 는 UI 이벤트 1개 구독 + 데이터 행 1개라 M1 비용이 사실상 0이므로 2개로 확정했다. 나머지 축소(썸네일·턴테이블·온보딩 재구성 등)는 전부 수용. (코인 자체가 M2 로 밀리면서(결정 5) 두 과제의 보상은 "M2 소급 지급"으로 재정리 — §1·§4.)
